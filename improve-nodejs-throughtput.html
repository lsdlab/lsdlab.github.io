（译文）提高你的 Node.js 应用吞吐量-一次只做一个微小的优化

Posted by Chen Jian on 2017-03-15 10:30:00 Updated on 2017-03-15 10:30:00

原文： Improve Your Node.js App Throughput One Micro-optimization at a Time

要点

  • 通过分组或批量写入以减少系统调用的次数。
  • 考虑分发和清除应用中的定时器的开销。
  • CPU 分析器能给你有用的信息，但不能给告诉你全部过程
  • 对 ECMAScript 的高级功能保持警惕，特别是在你没有使用最新的 JavaScript 引擎或者转码器的时候
  • 控制你的依赖树，对依赖进行性能测试。

为了提高一个涉及 IO 的应用性能，你应该理解 CPU 周期如何开销，更重要的是，如何阻止应用中更高程度的并行计算。

当聚焦在提高 Apache Cassandra 的 DataStax Node.js 驱动性能的时候，我积累了一些经验并分享在这篇文章中，尝试总结导致应用的吞吐量下降的最重要因素。

背景

Node.js 使用 JavaScript V8 引擎，将 JavaScript 代码编译成机器码然后当作原生代码运行。V8 引擎使用三个部件以同时达到 最短启动时间和最高性能：

  1. 一个通用的编译器将 JavaScript 代码尽可能快地编译成机器码。
  2. 一个运行时分析器记录运行每一段代码需要的时间，识别代码是否值得优化。
  3. 一个优化编译器尝试优化运行时分析器识别出来的代码，当优化编译器的假设结果被过度优化时，优化编译器也支持反优化。

优化编译器将达到最佳性能，但并不是所有的 JavaScript 代码都能被选中优化，有一些代码模式会被优化编译器拒绝优化。

你可以使用 this ticket from Google Chrome DevTools team 作为指南，用可能的变通方法优化被 V8 引擎拒绝优化的代码模式，包括一些示例：

  • 带有 try-catch 异常捕捉语句的方法。
  • 使用 arguments 给参数赋值的语句。

尽管优化编译器将使你的代码运行得十分快，但一个 IO 密集型程序的大多数性能提升都围绕这如何记录指令以及使用更低廉的调用，以允许每秒钟内进行更多操作，也将是一下章节要介绍的。

性能测试

为了发现优化能够影响到最大的用户数量，定义一个普通使用情况下的性能测试需要的负载量非常重要，需要模拟真实世界的使用量。

首先需要测量你的 API 的吞吐量和延迟。你可以给每个内部方法做性能测试，如果你需要更详细的信息，使用 process.hrtime() 获得更高准确率的实时时间和执行的需要的时间。

你应该尝试在项目早期创建一些有限制并且符合现实情况的性能测试，从测量方法调用的吞吐量开始，然后到增加更多的全面信息比如延迟。

CPU 分析器

Node.js 生态系统有几个 CPU 分析器，Node.js 提供了一个开箱即用并且对于大多数情况够用的分析器。Node.js 内置分析器 利用的 V8 内置分析器 的优势，在执行期间以规则的时间间隔对堆栈进行采样。你能够通过 —prof 标识运行代码生成 V8 tick 文件。

然后你可以使用 --prof-process 标识处理分析会话，聚合信息将其转换为可读的形式：

$ node --prof-process isolate-0xnnnnnnnnnnnn-v8.log > processed.txt

在编辑器中打开处理好的文本你将看到一些被分成章节的信息。

在文件中查找 “Summary” 部分，如下所示：

[Summary]:

ticks total nonlib name

20109 41.2% 45.7% JavaScript

23548 48.3% 53.5% C++

805 1.7% 1.8% GC

4774 9.8% Shared libraries

356 0.7% Unaccounted

值表示收集的样本中有多少发生在 JavaScript / C ++代码/垃圾收集器中，并且将根据你分析的代码而有所不同。 然后，你可以导航到文件的相应子部分（例如[JavaScript]，[C ++]，...），按样本出现的顺序查看详细信息。

在剖析会话的已处理文件中有一个附加部分，[Bottom up（heavy）profile]，这是特别有用的。 它以树状结构提供关于每个函数的主调用者的信息。 以下面的片段为例：

223 32% LazyCompile: *function1 lib/file1.js:223:20

221 99% LazyCompile: ~function2 lib/file2.js:70:57

221 100% LazyCompile: *function3 /lib/file3.js:58:74

百分比显示特定调用者在其父调用总量中的份额， 函数名前面的星号表示用于优化函数所消耗的时间，而波形表示未优化函数。

在该示例中，function1 的调用的 99％ 由 function2 进行，根据剖析示例，function3 负责对 function2 的100％ 的调用。

CPU 分析会话和火焰图是有用的工具，以了解大多数时间内堆栈里面是什么，哪些方法花费 CPU 时间，以发现显而易见的问题。 但重要的是要了解它不会告诉你全部过程：你可能会阻止在你的应用中实现更高程度的并行，异步 IO 操作也将使其更难识别。

系统调用

Libuv 暴露了一个独立于平台的 API，Node.js 使用它来执行非阻塞 IO，你的应用 IO（套接字，文件系统...）最终将转换为系统调用。

在调度这些系统调用时存在明显的成本，你应该尝试通过分组/批处理写入来最小化系统调用的数量。

当使用套接字或文件流时，不是每次都发出写入命令， 你可以不时地缓冲和刷新数据。

你可以使用写入队列来处理和分组写入。写队列实现的逻辑应该是：

  • 虽然有项目要写，我们在窗口大小内
  • 将缓冲区推送到“写入列表”
  • 连接列表中的所有缓冲区，并将其写入。

你可以根据总缓冲区长度或自第一个项目的排队时间来定义窗口大小。定义窗口大小是单个写入的延迟和所有写入的平均延迟之间的权衡。你还应该考虑分组写入请求的最大数量和生成每个写入请求的开销。

你通常希望以千字节的顺序刷新写缓冲区。我们发现大约 8 千字节的关键点，但你的关键点可能不同。你可以在客户端驱动程序中查看完整的写队列实现。

由于较少的系统调用，分组或批处理写入将变成更高的吞吐量。

Node.js 定时器

Node.js 定时器 具有与 Web API 中的 窗口定时器 相同的 API，非常有用，易于调度/解调，并广泛用于整个生态系统。

因此，在应用程序中的任何给定时间可能存在大量的超时时间。

与其他 散列轮定时器 类似，Node.js 使用散列表和链表 来维护定时器实例。但是与其他轮定时器不同，它不是具有固定长度的哈希表，而是按持续时间对每个定时器列表进行锁定。

当存在密钥（存在具有相同持续时间的定时器）时，它作为O(1)操作被附加到桶。

当键不存在时，将创建一个存储桶，并将定时器附加到该存储桶。

记住这一点，你必须确保你重用现有的桶，尽量避免删除整个桶，并创建一个新的桶。例如，如果你使用滑动延迟，则应在清除 (clearTimeout()) 前创建新的超时(setTimeout())。

在我们的用例中，通过在移除前一个之前 调度空闲超时(心跳)，确保空闲超时的调度和解调是O(1)操作。

ECMAScript 功能

有几个高级 ECMAScript 功能，如果你关心性能，你应该避免使用。这样的功能的例子是： Function.prototype.bind(), Object.defineProperty() 和 Object.defineProperties()。

这些功能中大多数性能较差可以通过 JavaScript 引擎中的实现细节来解释，其中一些问题正在解决，例如：V8 5.3 中的 Promise 性能改进 和 V8 5.4 中的 Function.prototype.bind 性能。

对于 ES2015 和 ESNext 中的新语言功能，你应特别小心，这可能会比他们的 ECMAScript 5 对应版本慢得多。six-speed project 网站跟踪不同 JavaScript 引擎的当前性能。此外，当你无法从现有基准中找到结论性结果时，你可以对不同方法进行小的性能测试。

V8 团队正在努力提高新语言功能的性能，最终达到原始水平，他们的优化功能介绍与 ES2015 的工作通过一个 性能计划 协调，其中 V8 团队收集需要改进的领域，并提出了解决这些问题的设计文件。

你可以在 博客上跟踪 V8 实现的进度，但是考虑到这些改进可能需要很长时间才能进入 Node.js 的长期支持（LTS）版本：哪个 V8 版本进入 Node.js 的主要版本根据 LTS 计划，通常在从主控切割分支之前决定。你将不得不等待 6-12 个月，一个新的主要/次要版本的 V8 引擎被包括在 Node.js 运行时。

Node.js 的主要版本中的新版本 只会更新 V8 引擎的修补程序。

依赖

Node.js 运行时提供了完整的 IO 库，但是 ECMAScript 标准也提供了一些内置的类型，有时候你需要依赖外部包进行一些基础操作。

对于流行的外部包也不能保证其实现是最高效的，Node.js 生态系统十分庞大，有一些第三方包中的一些方法通常你也能够自己实现。

你应该在重新发明轮子和控制第三方依赖包带来的性能影响取得均衡。

尽可能避免添加一个新的依赖，不要过分相信你的依赖，但也有个例外就是如果第三方包的开发者进行了非常可靠的测试，比如 bluebird 这个包。

在我们的用例中，async 会影响到请求的延迟，我们在代码库中大量使用 async.series(), async.waterfall() 和 async.whilst。控制流库作为一个横向关注点，非常难以区分是其否是性能低下的源头。async 作为非常流行的模块之一，也正因为如此，问题也被确认。甚至有直接替换的库比如 neo-async，这个库也十分快速并且发布了性能测试。

总结

尽管文中描述的一些优化方法能够适用于其他的语言，有一些则是为配合 JavaScript 引擎和核型库以及 Node.js 生态系统特有的优化方法。

在进行了一些优化后，我们的驱动的性能测试结果显示提高了两倍的吞吐量。

考虑到我们的我们的 Node.js 代码运行在一个单独的线程中，我们的应用如何使用 CPU 周期以及调度指令十分重要，允许我们能够支持更高程度的并行，提高总体的吞吐量。
